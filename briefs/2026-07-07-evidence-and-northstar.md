### 目的

### 方案

1. **Seed 一条示范 draft 进 bundle**（不是 runtime store，避免被 50 条 ring buffer 挤掉；`local-json-repo` 合并时可见）：
   - 客户：Jensen book 里选一个非首屏客户（避免与现场演示的客户冲突）。
   - 时间跨度 3–4 天，完整链路：`created`（Jensen，Email / Client Review Pack）→ `rejected`（Sofia，note 写具体退回理由，如 "Remove the yield projection line — replace with review-meeting invitation"）→ `edited`（Jensen）→ `approved`（Sofia）→ `sent`。

2. **History 时间线组件**：在 draft 审阅视图（manager 的 Read draft 模式）和 Sent 条目详情中加 "History" 区：垂直时间线，每步 = 状态徽章 + 操作人 + 相对时间 + note。状态色用现有 tone token，不引入新色。
3. Demo script 加半页："打开这条历史 draft，从退回意见看到修改再到发送——这就是审计重建"。它同时是 Task 2 导出的标准演示样本。

## Task 4（P1）：北极星指标 —— Governed touches & coverage（明确不做 ROI 换算）

### 定义（银行真实业务口径）

- **Governed touches / week**：经过 Beacon 流程发出的客户触达（draft sent + 记录的 call/meeting），周口径。
- **Coverage 90d**：book 内 90 天有过触达的客户占比。`lib/domain/governance.ts` 的 `contactedIn90dPct` 已经实现，只是 UI 没显示。

这两个指标对应银行的 service-model 承诺（分层客户的触达频率要求）和 KYC/review 义务——是业务真实存在的管理口径。

### 呈现

1. RM workspace 加一个小 tile：本周 touches + 90d coverage %（个人）。
2. Manager 团队表：touches/wk 已有，补 coverage % 列；趋势可后续再加，本期不做图表。
3. 文案框架："coverage" / "touches"，动词用 prepare/serve。

### 明确禁止

- **不做任何金额或时间节省换算**：不出现 "saved X hours"、"worth $Y"、每 touch 定价一类文案（pricing model 另议，产品里不预设）。
- 本 Task **取代** `2026-07-07-presales-polish.md` Task I.2（ROI 条）——I.2 作废不做。Task I.1（evidence 点击跳转）与 I.3（demo preset）保留为 P2 待办，不在本 brief 范围。
